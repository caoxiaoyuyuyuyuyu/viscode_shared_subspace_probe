## Core Problem

When an LLM generates SVG, TikZ, or Asymptote code for the same visual concept (e.g., 'a red circle above a blue square'), do its hidden states encode a format-agnostic representation of the rendered scene, or are the representations format-specific? And — the predictive question that makes this matter — does the geometry of that representation (per-layer cross-format probe transferability) actually predict which format pairs the model will behaviorally transfer between? We test this by training linear probes on hidden states from one format's generations to decode rendered-image attributes (color, shape count, layout, spatial relations, stroke/fill parity, scene class) and measuring whether they generalize to hidden states from a different format in the same model.

## Opportunity / Why Now

Visual code generation has exploded in 2025-2026 (LLM4SVG, StarVector, Chat2SVG, VisCoder2, IntroSVG) but the field is almost entirely phenomenological: papers report benchmark numbers and occasionally train multi-language models, yet nobody has asked what visual structure actually lives inside the LLM's residual stream during code generation. Concurrently, code-LLM interpretability (Neuron-Guided Interpretation 2512.19980; SAE Code-Correctness 2510.02917) has begun probing hidden states but exclusively for code-syntactic and semantic features in C++/Java/Python. The missing axis is *visual grounding*: does the same residual stream that produces `<circle cx=...>` or `\draw (0,0) circle` carry a format-agnostic encoding of the *rendered image* — color, shape count, spatial layout? A training-free linear-probe study answers this first-order mechanistic question with inference + CPU probe fitting only. Crucially, by correlating probe-transfer accuracy with behavioral transfer, we promote the result from descriptive ('a shared subspace exists') to predictive ('representation geometry forecasts which format pairs will transfer'), giving multi-format-training designers (à la VisCoder2) a cheap diagnostic for which language combinations are worth bundling.

## Landscape (SOTA & Limitations)

**Visual-code generation (the models we dissect).** VisCoder2 (arXiv 2510.23642, Oct 2025) trains a 679K multi-language SFT model over 12 visualization languages including SVG/TikZ/Asymptote/Matplotlib/LilyPond and introduces VisPlotBench; its evaluation is purely behavioral (execution pass rate, VLM-judge scores) with no analysis of internal representations. LLM4SVG (CVPR 2025), Chat2SVG (CVPR 2025), and IntroSVG (2026) are single-format SVG specialists. SVGenius (ACM MM 2025), vTikZ (EASE 2025), VGBench (EMNLP 2024), and VisPlotBench (2025) provide ready-made caption/code/render evaluation suites. **Code-LLM interpretability (the methodologically adjacent gap).** Two 2025 papers come closest: (a) Neuron-Guided Interpretation of Code LLMs (arXiv 2512.19980) uses neuron-selectivity and RSA to find 'concept layers' shared across C++/Java/Python/Go/JavaScript on Llama-3.1-8B and Qwen2.5-Coder-32B; (b) Mechanistic Interpretability of Code Correctness via SAEs (arXiv 2510.02917) trains SAEs on code-LLM residual streams to find correctness-predicting directions. Both probe *textual code* features (syntax, semantics, correctness); neither touches rendered-image attributes or visual code formats. **Our axis is orthogonal.** Where 2512.19980 asks 'is `for-loop` represented the same way in Python and Go residual streams', we ask 'is `red circle above blue square` represented the same way in SVG and TikZ residual streams' — the probe target is a property of the *rendered output image*, not of the code token sequence. To our knowledge no prior work has decoded image-grounded attributes from code-generation hidden states, in any format. **General linear-probe and representation-similarity foundations** — structural probes (Hewitt & Manning 2019), CKA (Kornblith et al. 2019), the linear representation hypothesis (Park et al. 2024), propositional probes (2024) — give us the methodological tools. The structural gap is mechanistic *and* visually grounded, not behavioral.

## Proposed Approach

Pure training-free + one tiny ablation, on a single RTX PRO 6000 (96GB). **Three shape-oriented formats: SVG, TikZ, Asymptote** (Vega-Lite dropped — declarative chart-spec JSON makes color/shape/layout attributes ill-posed; Asymptote is a true shape-drawing DSL with coverage in the VisCoder2 corpus). No fresh scraping — use existing HF datasets: SVGX-Core (SVG), vTikZ + DaTikZv2 training splits (TikZ), VisCoder2's Asymptote split (Asymptote). Sample 3K paired (caption, code, rendered PNG) triples per format, with all rendered images pre-computed before probing begins. **Models evaluated off-the-shelf, all text-to-code with comparable residual-stream taps:** (a) Qwen3-Coder-7B (generic code baseline), (b) Qwen2.5-Coder-7B (second generic baseline, replaces StarVector — StarVector dropped because its multimodal vision-encoder→LLM-decoder stack makes residual-stream probes architecturally incomparable to text-only models), (c) LLM4SVG-7B (SVG specialist), (d) VisCoder2-7B (multi-format SFT). All served in 4-bit via vLLM. **Stage A — behavioral transfer matrix (≈14 GPU-h)**: for each (model, format) pair, run zero-shot + 3-shot generation on 500 held-out captions per format from SVGenius/vTikZ/VisPlotBench and score with format-validity (parses & renders) + CLIP ViT-L similarity to a caption-derived reference. Produces a 4×3 model×format behavioral table. *Pre-registered control:* the LLM4SVG row will be ~0 on TikZ/Asymptote — recorded as expected single-specialist behavior, not a finding. *Budget calc:* 4 models × 3 formats × 500 prompts × 2 shot regimes = 12K generations × ~800 output tokens / ~450 tok/s effective vLLM-batched throughput = ~5.9 GPU-h generation + ~7 GPU-h rendering+CLIP scoring overhead ≈ 13 GPU-h. **Stage B — hidden-state probes with expanded attribute battery (≈22 GPU-h)**: during generation, cache last-token residual stream at layers {8,16,20,24,28,32} for ~2K prompts per (model, format). Train lightweight logistic/ridge probes (sklearn, CPU) to predict **six** visual attributes auto-extracted from the rendered PNG: (i) dominant color histogram bucket (8-way), (ii) shape count (CV contour count, capped 0-9), (iii) layout bbox coarse quadrant (4-way), (iv) **relative spatial relation between top-2 shapes** (above/below/left/right/inside, 5-way) — addresses the 'too coarse' critique with a compositional attribute, (v) **stroke/fill parity** (filled vs outline-only majority, binary), (vi) CLIP-image zero-shot class from a fixed 50-way taxonomy. Six attributes × six layers × four models × three formats × cross-format pairs gives a richer geometry than the original 4-attribute design. **Key experiment: cross-format probe transfer matrix** — train probe on SVG hidden states, test on TikZ hidden states from the same model; high cross-format probe accuracy ⇒ shared visual-program subspace at that layer. **Stage C — CKA / Procrustes alignment (≈4 GPU-h, CPU-heavy)**: linear CKA and Procrustes between per-format hidden-state matrices within each model, as an independent geometric signal from the probes. **Stage D — predictive claim (no extra GPU; pure analysis)**: compute Spearman ρ between per-layer cross-format probe-transfer accuracy (Stage B) and behavioral cross-format transfer scores (Stage A) over the (model × layer × format-pair) grid. *Headline outcome:* a positive correlation upgrades 'shared subspace exists' to 'representation geometry predicts which format pairs will transfer behaviorally', a NeurIPS-shaped predictive claim; a null correlation is also publishable as a probe-behavior decoupling finding. **Stage E — one controlled QLoRA ablation (≈14 GPU-h)**: take Qwen3-Coder-7B and QLoRA-SFT on 20K SVG samples only (rank 16, bf16 base + 4-bit, ≈9 GPU-h training). Re-run Stages A-D on the fine-tuned model. Question: does single-format SFT (i) improve or destroy cross-format probe transfer, (ii) expand or collapse the shared subspace, (iii) preserve or break the Stage-D predictive correlation? Causally links training to representation geometry. **Total budget: ≈53 GPU-h + ~17h eval/probe headroom ≤ 70 GPU-h.** No RL, no Claude/VLM judge — CLIP + renderer + CV only. **Differentiation summary:** vs VisCoder2 — they train, we dissect; vs 2512.19980 / 2510.02917 — they probe code-syntactic/correctness features in textual languages, we probe rendered-image attributes in visual code formats; vs all SVG specialists — we ask whether their representations generalize beyond their training format and predict transfer from geometry alone.
